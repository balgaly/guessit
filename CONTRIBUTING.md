## Prerequisites

Before you begin, make sure you have the following installed:

- **Node.js 20+** ([download](https://nodejs.org/))
- **pnpm** (install via `npm install -g pnpm`)
- **Git** ([download](https://git-scm.com/))
- **GitHub CLI** (optional but recommended): `gh` ([installation](https://cli.github.com/))
- **Claude Code** (choose one):
  - **CLI:** `npm install -g @anthropic-ai/claude-code` ([docs](https://github.com/anthropics/claude-code))
  - **VSCode Extension:** Install from [VSCode Marketplace](https://marketplace.visualstudio.com/items?itemName=Anthropic.claude-code)
- **Supabase CLI:** `npm install -g supabase` ([docs](https://supabase.com/docs/guides/cli))

## Local Development Setup

### 1. Clone the Repository

```bash
git clone https://github.com/yourusername/guessit.git
cd guessit
```

### 2. Install Dependencies

```bash
pnpm install
```

### 3. Set Up Environment Variables

Create a `.env.local` file in the project root by copying the example:

```bash
cp .env.example .env.local
```

Edit `.env.local` and fill in the following values:

```bash
# Supabase (Local Development)
NEXT_PUBLIC_SUPABASE_URL=http://localhost:54321
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-local-anon-key  # Generated by `npx supabase start`

# Supabase (Optional: Cloud Project)
# NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
# NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key

# The Odds API (for odds-based scoring)
THE_ODDS_API_KEY=your-api-key  # Get from https://the-odds-api.com/

# NextAuth (for production)
NEXTAUTH_URL=http://localhost:3000
NEXTAUTH_SECRET=generate-with-openssl-rand-base64-32
```

**Where to get each key:**
- **Supabase Local Keys:** Run `npx supabase start` (see next step) — keys are printed in the console output
- **The Odds API Key:** Sign up at [the-odds-api.com](https://the-odds-api.com/) (free tier available)
- **NEXTAUTH_SECRET:** Generate with `openssl rand -base64 32`

### 4. Start Supabase Local Development Instance

```bash
npx supabase start
```

This command:
- Downloads Supabase Docker containers (first run takes a few minutes)
- Starts a local PostgreSQL database, Auth server, and Supabase Studio
- Prints your local `NEXT_PUBLIC_SUPABASE_ANON_KEY` and other credentials

Once it's running, you can access:
- **Supabase Studio:** http://localhost:54323 (database GUI)
- **API:** http://localhost:54321

To stop Supabase later, run `npx supabase stop`.

### 5. Configure Your MCP Keys (for Claude Code)

Create a `.mcp.json` file from the example template:

```bash
cp .mcp.json.example .mcp.json
```

Edit `.mcp.json` and replace placeholders with your actual API keys:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "your-github-pat"
      }
    }
  }
}
```

**Where to get your GitHub PAT:**
1. Go to [GitHub Settings > Developer Settings > Personal Access Tokens](https://github.com/settings/tokens)
2. Generate a new token (classic) with `repo`, `read:org`, and `user` scopes

**Note:** `.mcp.json` is gitignored — never commit your actual keys!

### 6. Run the Development Server

```bash
pnpm dev
```

Visit [http://localhost:3000](http://localhost:3000) to see the app. If you see the login page, everything is working!

## Understanding Our Agent/Skill Ecosystem

GuessIt uses **Claude Code agents and skills** to help developers work more efficiently. Think of them as specialized AI assistants for different parts of the codebase.

### What Are Agents?

Agents are defined in `.claude/agents/*.md` and handle complex, multi-step tasks. Each agent is an expert in a specific domain.

### What Are Skills?

Skills are defined in `.claude/skills/*/SKILL.md` and can be invoked with slash commands like `/backend-lead` or `/security-engineer`. They're shortcuts for common workflows.

### Quick Reference: Available Agents & Skills

| Slash Command | Domain | When to Use |
|---------------|--------|-------------|
| `/ceo-strategist` | Business Strategy | Feature prioritization, go-to-market, monetization |
| `/cto-architect` | System Architecture | Database schema, API design, tech stack decisions |
| `/product-manager` | Product Management | User stories, sprint planning, roadmap |
| `/devops-engineer` | Infrastructure | CI/CD, deployment, Vercel, Supabase config |
| `/frontend-lead` | Frontend (React/Next.js) | UI components, pages, Tailwind styling, RTL |
| `/backend-lead` | Backend (API/Database) | API routes, Supabase queries, business logic |
| `/security-engineer` | Security | Auth, RLS policies, input validation, OWASP |
| `/qa-engineer` | Testing | Writing tests, test strategy, bug reproduction |
| `/ux-designer` | UX/Design | User flows, accessibility, mobile UX |
| `/data-analyst` | Analytics | Metrics, KPIs, user behavior analysis |
| `/odds-api` | Odds Integration | Betting odds API, caching strategy |
| `/scoring-engine` | Scoring Logic | Prediction scoring, leaderboard calculation |
| `/deploy` | Deployment | Deploy to staging or production |
| `/onboarding` | Developer Onboarding | Help getting started (you're reading its output!) |
| `/backlog-manager` | Backlog Management | Create tickets, scan for TODOs, sprint planning |

### How to Use Agents

In Claude Code CLI or VSCode extension, type a slash command to invoke a skill:

```bash
# Example: Build a new React component
/frontend-lead create a predictions list component

# Example: Review security before launch
/security-engineer audit the auth flow for vulnerabilities

# Example: Plan a new feature
/product-manager write a spec for group invite links
```

## Development Workflow

### Picking a Task

1. Browse the backlog in [`docs/backlog/todo/`](./docs/backlog/todo/)
2. Pick a ticket that matches your skills and interests
3. Check the "Assigned Agent" field — use that agent to help you build it
4. Move the ticket to `docs/backlog/in-progress/` (or use `/backlog-manager move GI-XXX in-progress`)

### Branch Naming

Follow this convention:

- `feat/short-description` — New features
- `fix/short-description` — Bug fixes
- `chore/short-description` — Tooling, refactoring, docs

Example: `feat/group-invite-links`

### Conventional Commits

All commits must follow [Conventional Commits](https://www.conventionalcommits.org/):

```
feat: add group invite link generation
fix: prevent predictions after kickoff
chore: update dependencies
docs: improve CONTRIBUTING guide
test: add tests for scoring engine
```

### Creating a Pull Request

When your feature is ready:

1. **Push your branch:**
   ```bash
   git push origin feat/your-feature
   ```

2. **Create a PR:**
   ```bash
   gh pr create --title "feat: your feature" --body "Closes GI-XXX"
   ```

3. **Wait for reviews:**
   - The `security-engineer` agent will review for security issues
   - The `qa-engineer` agent will check test coverage
   - A human maintainer will review code quality and architecture

4. **Address feedback:**
   - Make changes in your branch
   - Push updates — the PR will update automatically

5. **Merge:**
   - Once approved, a maintainer will merge your PR
   - Move your ticket to `docs/backlog/done/`

## Code Review Process

Every PR goes through automated and human review:

### Automated Reviews
1. **CI Checks:** Tests, linting, type-checking (via GitHub Actions)
2. **Security Agent:** The `/security-engineer` agent scans for vulnerabilities
3. **QA Agent:** The `/qa-engineer` agent checks test coverage

### Human Review
A maintainer will review:
- Code quality and readability
- Adherence to architecture principles (see [CLAUDE.md](./CLAUDE.md))
- Proper use of TypeScript, React patterns, and Supabase
- Security best practices (RLS policies, input validation)

### Review Checklist
Before requesting review, ensure:
- [ ] All tests pass locally (`pnpm test`)
- [ ] Code follows our conventions (see [CLAUDE.md](./CLAUDE.md))
- [ ] No `any` types in TypeScript
- [ ] All API routes validate input with Zod
- [ ] Database queries use RLS policies (never bypass)
- [ ] Commits follow conventional commit format
- [ ] PR description references the ticket (e.g., "Closes GI-042")

## Your First Task

Let's walk through contributing a simple feature end-to-end.

### Step 1: Pick a Ticket

Browse [`docs/backlog/todo/`](./docs/backlog/todo/) and find a ticket labeled `priority:low` or `good-first-issue`. For example, let's say you pick `GI-015-health-check-endpoint.md`.

### Step 2: Move the Ticket

```bash
# Using the backlog-manager skill:
/backlog-manager move GI-015 in-progress

# Or manually:
mv docs/backlog/todo/GI-015-health-check-endpoint.md docs/backlog/in-progress/
```

### Step 3: Create a Branch

```bash
git checkout -b feat/health-check-endpoint
```

### Step 4: Build the Feature

Open Claude Code and invoke the appropriate agent:

```bash
# Since this is an API route, use the backend-lead agent:
/backend-lead implement the health check endpoint from ticket GI-015
```

The agent will:
- Read the ticket to understand requirements
- Create the API route file
- Write tests
- Validate that everything works

### Step 5: Commit Your Changes

```bash
git add .
git commit -m "feat: add health check endpoint for monitoring

Closes GI-015"
```

### Step 6: Push and Create a PR

```bash
git push origin feat/health-check-endpoint
gh pr create --title "feat: add health check endpoint" --body "Closes GI-015"
```

### Step 7: Wait for Review

Your PR will be reviewed by:
1. GitHub Actions (CI tests)
2. The `security-engineer` agent
3. A human maintainer

### Step 8: Merge and Celebrate! 🎉

Once approved and merged:
```bash
/backlog-manager move GI-015 done
```

Congratulations, you've made your first contribution!

## Getting Help

### Stuck on Setup?

Use the onboarding skill:
```bash
/onboarding help me troubleshoot my Supabase connection
```

### Need Guidance on a Feature?

Ask the appropriate agent:
```bash
/frontend-lead how should I structure the predictions form component?
```

- **Bugs:** Open a [GitHub Issue](https://github.com/yourusername/guessit/issues/new)
